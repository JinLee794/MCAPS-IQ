# Proof Orchestration

## When to Use

An opportunity has been qualified (Stage 1 → 2) and the customer needs technical validation before committing. The SE leads proof execution.

## Steps

1. **Define proof scope** — What specific question does this proof answer? Document in a meeting note.
2. **Set acceptance criteria** — Explicit, measurable criteria the customer agrees to. Add to customer file.
3. **Identify environment prerequisites** — What does the customer need to provide? What do we provision?
4. **Build timeline** — Realistic milestones with buffer. Create CRM tasks.
5. **Execute** — Deliver the proof. Log key findings in meeting notes.
6. **Capture outcome** — Did it pass acceptance criteria? Document evidence.
7. **Handoff artifacts** — Architecture decisions, configuration, and results ready for CSA if moving to delivery.

## Expected Outcome

- Proof completed with documented pass/fail against acceptance criteria
- Customer file updated with proof results
- Handoff-ready artifacts for next stage

## Common Pitfalls

- Scope creep — proof expands beyond original question
- No acceptance criteria — success becomes subjective
- Results not documented — value delivered but never captured
